## Core Metrics Framework

Every experiment measures performance across 4 dimensions:

### 1. Awareness (Reach)
- How many people saw the experiment?
- Did it reach our target audience?

### 2. Engagement (Interaction)
- Did people interact with the content?
- Quality of engagement (depth vs surface)

### 3. Conversion (Action)
- Did people take the desired action?
- Visit repo, star, sign up, clone?

### 4. Quality (Audience Fit)
- Are the engaged users our target audience?
- Technical founders, AI practitioners, not casual visitors

## Success Thresholds

**PASS**: 4/4 criteria met
**PARTIAL**: 3/4 criteria met (continue experiment with adjustments)
**FAIL**: 0-2/4 criteria met (pivot channel or approach)

## Quick-Win Experiments (QW-1 through QW-4)

### QW-1: Hacker News Launch

**Hypothesis**: "Rigorous R&D methodology" angle resonates with HN technical audience.

**Effort**: 4-6 hours (research, writing, timing)

#### Awareness Metrics
- Upvotes: Target 20+, Pass ≥15
- Comments: Target 10+, Pass ≥8
- Unique visitors: Target 50+, Pass ≥30

#### Engagement Metrics
- Comment quality (technical discussion): Pass ≥60% comments on topic
- Time on page: Target 60s+, Pass ≥45s

#### Conversion Metrics
- GitHub stars: Target 5+, Pass ≥3
- Repo clones: Target 10+, Pass ≥5
- Website visits (if applicable): Target 20+, Pass ≥10

#### Quality Metrics
- Technical audience: Pass ≥70% of comments from technical users
- Relevant discussion: Pass ≥80% on methodology/experiments, not features

#### Success Scorecard
- [ ] Awareness: Upvotes ≥15 OR comments ≥8 OR visitors ≥30
- [ ] Engagement: ≥60% technical comments OR time on page ≥45s
- [ ] Conversion: Stars ≥3 OR clones ≥5 OR visits ≥10
- [ ] Quality: ≥70% technical audience OR ≥80% on-topic discussion

**PASS Condition**: 4/4 criteria met
**Decision Rule**: If PASS, repeat HN experiments monthly. If PARTIAL, adjust angle. If FAIL, pivot channel.

## Measurement Workflow

### Pre-Experiment (Before Launch)
1. Define hypothesis clearly
2. Set specific targets for each metric
3. Baseline current metrics (stars, traffic, etc.)
4. Choose measurement tools (GitHub analytics, Twitter Analytics, HN upvotes tracker)

### During Experiment (Live)
1. Monitor engagement in real-time
2. Respond to comments/questions (authentically, not spam)
3. Capture qualitative feedback (quotes, pain points)
4. Track immediate conversions (stars, signups, DMs)

### Post-Experiment (24-72 Hours After)
1. Compile all metrics
2. Fill out scorecard
3. Document learnings (what worked, what didn't)
4. Make go/no-go decision on channel

## Tools for Measurement

### GitHub
- Stars, watchers, forks (built-in)
- Clone traffic, traffic sources (GitHub Insights)
- Issue/PR activity (qualitative engagement)

### Hacker News
- Upvotes, comments (public)
- Comment quality (manual review)
- Referral traffic to GitHub (if tracking enabled)

### Twitter
- Impressions, engagements (Twitter Analytics)
- Profile visits, follows (Analytics)
- Thread metrics (thread completion rate)

### Discord
- Messages sent (server activity logs)
- DMs received (manual tracking)
- Community mentions (manual tracking)

### Website/Docs
- Pageviews, unique visitors (Google Analytics or similar)
- Time on page, bounce rate
- Referral sources

## Notes

- **Quality over quantity**: 50 highly-engaged technical users > 500 casual visitors
- **Pass threshold (4/5)**: Balances ambition with realism. Requires success across dimensions, not just one.
- **Iterate quickly**: If experiment FAILS, don't persist. Pivot or adjust within 1-2 weeks.
- **Document everything**: Even failed experiments teach us about our audience.
